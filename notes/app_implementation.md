# Webapp implementation plan

This document is the detailed plan for the *Currently* webapp: a mobile-first map showing tide and current stations with an interactive timeline scrubber. It assumes you have already read:

- `notes/calculating_primary_tides_and_currents.md` — the interpolation algorithm and timezone rules.
- `notes/gui_plan.md` — the original library shortlist.

This document overrides `gui_plan.md` where they disagree.

## 1. Goals and non-goals

**Goals (v1):**
- Mobile-first interactive map of every primary and secondary station in CHS Vol 5 (BC south coast / Strait of Georgia / Juan de Fuca).
- Timeline scrubber operating on **15-minute intervals** spanning the full available data range (currently one year, eventually multiple).
- Per-station icons that update in real time as the user scrubs to show interpolated tide height (m) and/or current speed/direction (knots, signed).
- Tap a station → table of that station's published extremes (HW/LW for tides; slack / max-flood / max-ebb for currents) for a relevant time window, plus a chart of the interpolated curve over the same window.
- Loads cleanly on a phone over 4G (cold load < 3 s to interactive). Smooth (60 fps) scrubbing on a mid-tier phone.
- Forward-compatible with additional CHS volumes (up to 7 total) and additional years (released annually in late autumn) without code changes — only data drops.
- No backend. 100% static SPA.

**Non-goals (v1):**
- Live data (NOAA/CHS realtime feeds, AIS, weather).
- Authentication, accounts, server-side state.
- Offline mode / installable PWA. (Architecture leaves this trivial to add later.)
- Cross-volume views (each session views one volume at a time).
- Centimetre-accurate harmonic predictions. The sinusoidal-between-extremes interpolator is sufficient for visualisation.

## 2. Architectural summary

| Layer | Choice | Notes |
|---|---|---|
| Map renderer | **MapLibre GL JS** | GPU vector tiles. Free fork of Mapbox GL. |
| Overlay rendering | **Native MapLibre symbol + circle layers**, GeoJSON-source-driven | At max ~900 stations across all volumes, this is fast enough that deck.gl is unjustified weight. |
| Basemap tiles | **PMTiles** (single static file, BC-coast OSM extract) | No tile server, no API key, range-request streaming. Easy to make offline-capable later. |
| Station detail chart | **uPlot** | ~40 KB gz, sub-millisecond render at our data sizes. |
| App framework | **Preact + Vite + TypeScript** | ~4 KB gz; React-compatible hooks/JSX with full tree-shaking. |
| State | **`@preact/signals`** (~1 KB gz) | Minimal observable store; no Redux/Zustand. |
| Time scrubber | **Plain `<input type="range">`** styled for fat thumbs | Mobile-native, accessible, zero deps. |
| CSS | Hand-rolled with CSS variables + a small reset | Avoid Tailwind weight for a small app. |
| Hosting | Static file host (Cloudflare Pages, GitHub Pages, S3+CloudFront, Netlify) | Free tier sufficient. |
| Dev server | Vite | HMR, instant rebuilds. |

**Approximate bundle size**: ~320 KB gz total — MapLibre ~250, uPlot ~40, Preact + signals ~5, app code ~25.

## 4. Data: source files and manifest

### 4.1 The four JSON files per (year, volume)

Already produced by `read_tct.py`:

- `{year}_tct_tidal_primary_stations.json` → `data/{year}/vol{N}/tidal_primary.json`
- `{year}_tct_tidal_secondary_stations.json` → `data/{year}/vol{N}/tidal_secondary.json`
- `{year}_tct_current_primary_stations.json` → `data/{year}/vol{N}/current_primary.json`
- `{year}_tct_current_secondary_stations.json` → `data/{year}/vol{N}/current_secondary.json`

Their JSON shapes are described in `calculating_primary_tides_and_currents.md` §"TypeScript implementation" and need no changes.

### 4.2 The manifest

A single file at a stable URL, regenerated by `build_manifest.py` after every data drop.

```json
{
  "generated_at": "2026-11-15T18:22:00Z",
  "volumes": {
    "5": {
      "name": "Juan de Fuca & Strait of Georgia",
      "years": [
        {
          "year": 2026,
          "tides_primary":     "2026/vol5/tidal_primary.abc123.json",
          "tides_secondary":   "2026/vol5/tidal_secondary.def456.json",
          "currents_primary":  "2026/vol5/current_primary.789abc.json",
          "currents_secondary":"2026/vol5/current_secondary.012def.json",
          "first_extreme_utc": "2026-01-01T01:24:00Z",
          "last_extreme_utc":  "2027-01-01T05:33:00Z"
        }
      ]
    }
  }
}
```

`first_extreme_utc` / `last_extreme_utc` are computed by `build_manifest.py` while it scans the JSONs. The app uses them to set the scrubber range without having to load every file up-front for that purpose.

Filenames embed an 8-char content hash (e.g. `tidal_primary.abc123.json`) so they can be cached forever.

### 4.3 Caching policy (load-bearing — get this right)

| Asset | Cache header | Reason |
|---|---|---|
| `manifest.json` | `Cache-Control: no-cache, must-revalidate` | Must reflect new data drops within one page load. |
| `data/**/*.{hash}.json` | `Cache-Control: public, max-age=31536000, immutable` | Filename changes when content changes; safe to cache forever. |
| JS/CSS bundle (Vite-hashed) | `public, max-age=31536000, immutable` | Same. |
| `tiles/bc.pmtiles` | `public, max-age=2592000` (30 d) | Basemap rarely changes; we don't hash it. |
| `index.html` | `no-cache` | Must always fetch newest hashed bundle references. |

Configure these via your host's headers config (Cloudflare Pages: `_headers` file; Netlify: `_headers`; S3: per-object metadata). Verify in DevTools after first deploy.

### 4.4 `build_manifest.py` (new — sketch)

```python
# Scan web/public/data/, find every (year, vol) directory, content-hash and rename
# the four JSONs in place, compute first/last extreme by parsing each JSON, write
# web/public/data/manifest.json. Idempotent. Called from process_tct.sh after
# read_tct.py finishes.
```

Implementation notes:
- Use `hashlib.sha256(file_bytes).hexdigest()[:8]` for the content hash.
- Don't re-hash files whose hash already matches their filename — keep it idempotent.
- Volume is currently always `5`; encode it explicitly in the path so the script generalises later.
- First/last extreme: walk `days[].readings[].time` (tides) and `days[].events[].time` (currents); convert each to UTC ms via the per-station `utc_offset`; track min/max.

### 4.5 Extending `process_tct.sh`

Add one line at the end:

```bash
python3 build_manifest.py --data-dir web/public/data
```

Re-running the script for the same year is safe (idempotent) and produces no diff if nothing changed.

## 5. Year handling — multi-year merge with no UI seams

Detailed in our prior planning. Summary:

1. **Manifest discovery**: `data/manifest.ts` fetches `manifest.json` once at startup. The app's known years/volumes come from this — never hardcoded.
2. **Eager-load all years of the active volume.** For Vol 5, all years total ~5 MB raw / ~1.5 MB gz per year. Two or three coexisting years is comfortable.
3. **Per-station merge**: stations are identified by **`index_no`** (CHS official ID, stable across years). For each `index_no`:
   - Concatenate `Extreme[]` from every loaded year for that station.
   - Sort by `t` ascending.
   - The sinusoidal interpolator from `calculating_primary_tides_and_currents.md` works unchanged.
4. **Stations may differ across years**: a station in 2026 but not 2027 simply has no 2027 extremes — `valueAt` returns null past Dec 31 2026 for that station. A station added in 2027 has no 2026 extremes — same outcome on the other side. Both correct.
5. **Metadata conflicts**: if a station exists in both years with slightly different `latitude` / `longitude` / `flood_direction_true`, **prefer the latest year's metadata.**
6. **Scrubber range** = union of `[first_extreme_utc, last_extreme_utc]` across all loaded years for the active volume.
7. **Default scrubber position** = `Date.now()`, snapped to the nearest 15-min grid, clamped to the loaded range. If clamping was needed (i.e. "now" is outside the range), show the `Banner` ("2026 data ends Dec 31; 2027 not yet published").
8. **Year transitions are invisible**: scrubbing across midnight Jan 1 looks identical to scrubbing across any other midnight; the merged extreme array doesn't know about year boundaries.

## 6. Map rendering

### 6.1 MapLibre setup

- Initialise on a `<div id="map">` filling the viewport (minus the scrubber and any overlay UI).
- Style: a minimal vector style pointing at the local `bc.pmtiles` via the `pmtiles://` protocol (registered with [`protomaps/PMTiles`](https://github.com/protomaps/PMTiles) JS library).
- Initial bounds: derived from the union envelope of all visible stations on first load. Fit with ~10% padding.
- Disable rotation/pitch by default (3D tilt is a v2 nice-to-have for current arrows; not core).
- Touch handling: MapLibre's defaults are correct; do not override.

### 6.2 Station GeoJSON source

A single `FeatureCollection`, kept in memory, with one `Feature` per station (regardless of year). Properties include:

```ts
type StationFeature = {
  type: "Feature";
  geometry: { type: "Point"; coordinates: [lon: number, lat: number] };
  properties: {
    station_id: number;            // CHS index_no (unique key)
    name: string;
    kind: "tide-primary" | "tide-secondary" | "current-primary" | "current-secondary";
    flood_dir: number | null;      // for current stations only
    ebb_dir: number | null;
    // Mutable per-frame fields (updated in place during scrub):
    value: number | null;          // metres for tides, knots (signed) for currents
    weak: 0 | 1;                   // 1 if currently within a weak/variable max segment
    rotation: number;              // for current arrows: flood_dir if value>0 else ebb_dir
  };
};
```

The same FeatureCollection object reference is reused across scrub frames. We mutate `properties.value` / `.rotation` / `.weak` in place, then call `source.setData(fc)`. MapLibre internally re-tessellates on each call; passing the same object reference means no unnecessary JSON re-parse cost.

### 6.3 Layers

Four layers, drawn in this paint order (bottom to top):

1. **`tide-secondary-layer`** — small circle, opacity tied to value-presence.
2. **`tide-primary-layer`** — larger circle, distinct stroke.
3. **`current-secondary-layer`** — small arrow symbol (sprite), `icon-rotate: ['get', 'rotation']`.
4. **`current-primary-layer`** — large arrow symbol, same rotation expression.

Tide circles encode height via fill colour (a diverging blue→white→red ramp around the station's mean). Current arrows encode signed knots via icon size + colour intensity (saturated for max, faded toward zero). Weak/variable maxes render with a hatched/hollow icon variant — driven by an `icon-image` expression that branches on `weak`.

All visual encodings are MapLibre **paint-property expressions** reading `properties.*` — no per-frame layer re-creation. The only per-frame work is mutating the FeatureCollection's properties and calling `source.setData()`.

### 6.4 Sprites

- One sprite sheet (PNG + JSON) compiled at build time from a small set of SVGs:
  - `arrow.png`, `arrow-weak.png` — current arrows (rotated by paint expression).
  - `tide-up.png`, `tide-down.png` — for the tide chevron variant if we want it (optional; circles are simpler for v1).
- Place the SVG sources in `web/src/map/icons/`; a tiny build script (or even just `spritezero-cli`) produces the sprite at `public/sprites/icons.{json,png}` referenced from the map style.

### 6.5 Tap-to-select interaction

- `map.on('click', layerIds, e)` — find the topmost station feature, set `selectedStationId` signal.
- On phones, also accept long-press to disambiguate touch-to-pan vs tap-to-select (MapLibre's default click-vs-drag threshold handles this without extra code).
- A station ring/halo paint expression highlights `properties.station_id === selectedStationId`.

## 7. Interpolation module

Code lifted directly from `calculating_primary_tides_and_currents.md` §"TypeScript implementation". Lives in `src/interp/`.

- `extremes.ts` exports `tideExtremes`, `currentExtremes`, `stationTimeToUtcMs`.
- `valueAt.ts` exports `valueAt(extremes, t)`.

Per-station extreme arrays are computed **once per station per app session**, the first time the station is needed for rendering, and cached on the in-memory station record. After multi-year merge they are sorted by `t`.

**Per-frame optimisation (deferred until profiling demands it):** add a per-station "last-segment cache" — store the index `i` such that `extremes[i].t <= t < extremes[i+1].t` from the previous frame. Most frames only advance by one 15-min step, so the cached index is still correct or one off; check that first, fall back to binary search otherwise. Turns the lookup into amortised O(1).

### 7.1 Secondary stations

Out of scope for this document — see the dedicated note that will be written when we implement them. Architectural hooks needed in v1:

- `data/loader.ts` already loads the secondary JSONs.
- The merge step accepts a station-extremes-builder function, so when secondary support arrives, only the builder changes — no plumbing changes.
- The map already has `tide-secondary-layer` and `current-secondary-layer`; until the secondary interpolator exists they render as muted "dot, no value yet" markers.

## 9. Data loader

`src/data/loader.ts`:

```ts
type LoadedYear = {
  year: number;
  tidesPrimary: TideStation[];
  tidesSecondary: TideSecondaryStation[];
  currentsPrimary: CurrentStation[];
  currentsSecondary: CurrentSecondaryStation[];
};

type LoadedVolume = {
  volume: string;
  years: LoadedYear[];
  scrubberRangeMs: { min: number; max: number };
  // Per-station merged extreme arrays, keyed by index_no:
  tideExtremesById: Map<number, Extreme[]>;
  currentExtremesById: Map<number, Extreme[]>;
  // Authoritative metadata per station (latest year wins):
  stationsById: Map<number, StationMeta>;
};
```

Steps:

1. Fetch `manifest.json` (no-cache).
2. For the active volume, fetch all years' four JSONs in parallel (`Promise.all`).
3. For each station across all years, compute and merge extremes (§5.3).
4. Build the map's GeoJSON FeatureCollection from `stationsById`.
5. Set `loadedVolume.value` to trigger reactive re-render.

## 10. UI / UX

### 10.1 Layout (responsive)

- **Mobile portrait (≤ 768 px wide)**: full-bleed map; scrubber pinned to bottom (~88 px tall, including time label and step buttons); tapping a station slides up a bottom sheet covering ~50% of the viewport with the chart on top, table below; swipe-down to dismiss.
- **Tablet / desktop (> 768 px)**: full-bleed map; scrubber pinned to bottom; station detail appears as a fixed right-side panel (~360 px wide) instead of a bottom sheet.
- **One CSS file** with media queries; no separate mobile/desktop component trees.
- All hit targets ≥ 44 × 44 px.

### 10.2 Scrubber

- `<input type="range" min={rangeMs.min} max={rangeMs.max} step={15*60*1000} value={scrubberMs.value}>`.
- Adjacent buttons: «−15m», «−1h», «−1d», «Now», «+1d», «+1h», «+15m». "Now" snaps to current 15-min grid and clamps.
- Time label: `Intl.DateTimeFormat("en-CA", { timeZone: "America/Vancouver", … })` — shows e.g. `Sun, Apr 27 · 14:30 PDT`. Browser handles BC's permanent UTC-7 from Nov 1 2026 automatically.
- **Coalesce input events to rAF**: store latest value in a ref; on each `requestAnimationFrame`, if changed, write it to the signal. Prevents 120-Hz touch event storms from triggering 120 map re-renders.
- Keyboard: arrow keys = ±15 min, Shift+arrow = ±1 h. Native `<input type=range>` supports this for free.

### 10.3 Station detail sheet

- Header: station name, type badge (tide/current, primary/secondary), lat/lon, distance from selected station optional.
- **Chart** (uPlot): plots interpolated value vs. time over `[scrubberMs − 12 h, scrubberMs + 12 h]`. Vertical line at `scrubberMs`. Dots at each published extreme. For current stations, a horizontal zero line and a flood/ebb hatched band background.
- **Table**: list of published extremes for the surrounding ±48 h window. Columns: time (local), kind (HW/LW or slack/max-flood/max-ebb), value (metres or knots). Weak/variable rows visually distinct.
- **Close button**: explicit, top-right; also dismissable by swipe-down on mobile or click-outside on desktop.

### 10.4 Banner

A small dismissable strip across the top, shown only when:
- Manifest fetch failed (offline or bad config).
- No data is available for the user's current wall-clock time within the active volume.
- Active volume's range ends in the past (e.g. it's Jan 5 2027 and only 2026 is loaded).

Wording for the third case: *"2026 data ends Dec 31. The 2027 tables haven't been added yet — scrub back to see prior data."*

### 10.5 Volume picker

Stubbed; hidden when the manifest contains exactly one volume. When 2+ volumes exist, a small `<select>` in a corner of the map; switching volumes triggers a re-load via `loader.ts` and resets `selectedStationId`.

## 11. Performance plan

**Per-frame budget** at 60 fps = 16.67 ms.

**The scrub frame's actual work:**
1. rAF callback reads latest scrubberMs (1 µs).
2. For each station whose extreme array exists: `valueAt(extremes, t)` — O(log n) ≈ 5 µs. With 900 stations: **~4.5 ms worst case**, more like ~1–2 ms typically.
3. Mutate FeatureCollection properties in place (negligible).
4. `source.setData(fc)` — MapLibre re-tessellates. For ~900 point features, **~2–3 ms**.
5. GPU draw — mostly off the main thread.

Total main-thread work per frame: well under 8 ms. Headroom for the rendering/compositor work in the remaining ~8 ms is comfortable.

**Optimisations applied up-front:**
- Single shared FeatureCollection object, mutated in place.
- rAF coalescing of scrubber input events.
- Station extreme arrays computed lazily per station, cached.
- Dead-simple binary search; no allocations in the hot path.
- All visual encodings as paint-property expressions, not per-frame style updates.

**Optimisations deferred** (apply only if profiling warrants):
- Last-segment-cache replacement for binary search.
- Web Worker for interpolation. Cost (postMessage + serialisation) likely exceeds savings for our N.
- Custom WebGL layer with a single VBO and a per-frame uniform array. Massive complexity for negligible gain at our N.
- Cull off-screen stations. Simple to add (`map.getBounds()` filter), only matters at the high-volume / high-zoom edge.

**Bundle size targets:**

| Chunk | Target | Notes |
|---|---|---|
| Vendor (MapLibre + Preact + signals + uPlot + PMTiles) | ~300 KB gz | Cached forever after first load. |
| App code | ~25 KB gz | |
| Initial Vol 5 data (one year) | ~1.5 MB gz | Largest blob; consider streaming display before all four files have parsed. |
| Basemap PMTiles | ~30–50 MB | Range-requested per visible tile; never fully downloaded. |

**Cold-load budget (4G phone, no cache):** target < 3 s to interactive — first map paint and first station icons rendered.

## 12. Build & deploy

### 12.1 Vite config highlights

- `base: "./"` for portable hosting.
- `build.target: "es2020"` — covers all phones < 5 years old.
- Asset filenames hashed by Vite default.
- A custom plugin or pre-build step to:
  - Copy `public/data/` through unchanged (Vite's default behaviour for `public/`).
  - Build the icon sprite from `src/map/icons/*.svg`.

### 12.2 Hosting

**Recommended**: Cloudflare Pages. Free tier, global CDN, simple `_headers` file for the cache rules in §4.3. Deploy via `wrangler pages deploy web/dist` or by pushing to a connected Git branch.

**Alternative**: GitHub Pages. Cache headers are not configurable per-file; the manifest will be served with a generic ~10-min cache, which is acceptable but not ideal. Use only if Cloudflare Pages is unavailable.

### 12.3 Annual data drop workflow

When the 2027 PDF is published (typically late autumn):

1. Download the PDF; place in repo root.
2. Run `process_tct.sh` (which now also runs `build_manifest.py`).
3. Verify the new files appear in `web/public/data/2027/vol5/` and that `manifest.json` lists 2027.
4. `git add web/public/data/2027 web/public/data/manifest.json && git commit && git push`.
5. CI/CD (or your manual deploy) publishes. Users see 2027 data on next page load.

No app code changes. No version bump. No app re-deploy required if the data is hosted on the same static host as the bundle.

## 13. Testing strategy

### 13.1 Unit tests (Vitest)

- `interp/valueAt`: the six sanity checks listed in `calculating_primary_tides_and_currents.md` §"Sanity checks for testing".
- `interp/stationTimeToUtcMs`: PST/PDT/permanent-PDT conversion cases.
- `data/merge`: multi-year merge produces a sorted Extreme[] across year boundaries; metadata-conflict resolution prefers latest year.
- `util/time`: 15-min snapping and clamping at range edges.

### 13.2 Integration / visual

- A simple Playwright script that:
  - Loads the app against a static snapshot of `manifest.json`.
  - Verifies the scrubber range matches the manifest's union range.
  - Drags the scrubber and asserts `getRequestCount` for `setData()` is rAF-bounded.
  - Taps a station and asserts the sheet appears with the expected extremes.

### 13.3 Real-device checks (manual)

Pre-ship checklist on a real iPhone and a real Android mid-tier:
- Cold load < 3 s on a throttled-4G profile.
- Scrubbing for 30 s shows no skipped frames in DevTools' Performance panel.
- Tapping a station never accidentally pans the map.
- Bottom sheet swipe-down dismiss feels native.

## 14. Open items intentionally deferred

- **Service worker / installable PWA**: easy to add later; not v1.
- **Offline mode**: requires SW + cached basemap PMTiles; not v1.
- **Particle/flow-field current animation** (windy.com style): requires a custom GL layer; out of scope for point-station data.
- **Cross-volume views**: one volume at a time in v1.
- **3D map tilt with depth-sorted current arrows**: visually nice; not v1.
- **Sharing a specific scrubber state via URL** (`?t=2026-04-27T14:30Z&station=7585`): trivial to add later via URL hash sync; not v1.
- **i18n (FR for Canadian audience)**: data is English-only from the parser today; strings are English-only in v1. Wrap UI strings in a tiny `t(key)` helper to make later i18n a one-week task.

## 15. Implementation order (suggested)

1. **Scaffold `web/`** with Vite + Preact + TypeScript. Verify dev server runs.

6. **Scrubber UI** + signals wiring; verify per-frame updates work and stay under budget.
7. **Tap-to-select + station detail sheet** with table.
8. **uPlot chart** in the sheet.
9. **Banner** for edge cases (no data for now, manifest fetch failure).
10. **Real-device perf pass** + bundle-size audit.
11. **Deploy to Cloudflare Pages** with the cache headers in §4.3.

Each step is independently shippable to a dev preview; the order minimises rework.
